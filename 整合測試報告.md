## 為何改用整合測試（而非瀏覽器 E2E）

原規劃用 `e2e-ui-test`（headless Chromium）錄影驗證,但本專案是 **pywebview 桌面
app**:前端 `index.html` 僅透過 `window.pywebview.api.*` 與 Python bridge 溝通,
該物件由桌面視窗執行時注入,且無對外 HTTP server。headless 瀏覽器載入頁面時
`window.pywebview` 不存在,35 個 API 全部呼叫不到,**無法驅動真實後端管線**。
故改用 pytest 直接呼叫 Bridge 方法,驗證真實的後端邏輯與把關。代價:沒有 UI 畫面
/ 錄影,且純前端渲染（toast、modal、徽章顏色）與需外部引擎的步驟不在此層涵蓋。

## 測試點來源與分類

交叉比對 `開發計畫.md`（PRD/設計）× 前端 35 個 js_api × Bridge 後端實作。
專案先前**無任何自有測試**,本次補上 43 條整合測試,聚焦可決定性的後端邏輯。

- **相容性發現**:pywebview app 不可用 headless 瀏覽器 E2E（見上）。此為方法論層面的
  重要結論,已改採整合測試。
- 另:稍早附帶修正的 onefile worker 啟動 bug（`sys.executable` → `NUITKA_ONEFILE_BINARY`,
  `src/bridge/_worker.py`）尚未被本套件涵蓋,因其僅在打包版觸發;建議重打包後手動驗證。
