# AI AudioFlow 開發計畫

> 本文件依據目前討論定案的架構整理而成，作為後續開發的依據。
> 最後更新：2026-06-17

---

## 1. 專案目標

把會議／通話音訊「錄製 → 轉譯 → 整理 → 後續處理」自動化。全鏈：
**穩定錄音 → 本地資料夾 → STT worker 轉譯 → 逐字稿 → openclaw 呼叫 LLM 整理 → 摘要 / 分發**。

設計原則：**各段以穩定合約解耦**（檔案資料夾 or OpenAI 相容 HTTP），每段可獨立重啟、獨立抽換。
逐字稿之後的「整理 / 摘要 / 分發」由 **openclaw（後續處理平台，獨立）** 負責；LLM 後端再包成一支**獨立的 LLM 服務**（§5.7），openclaw 不直接綁定 `claude -p`。

---

## 2. 定案架構

採**檔案系統交接**（非雲端 / 非 Make-Zapier）。所有元件在同一台機器（Windows + WSL2 + Docker Desktop）。

**核心決策：STT 與 openclaw 解耦。** 「錄音檔 → 逐字稿」由一支獨立 worker 處理，走 HTTP 直接打 whisper 容器；openclaw 的輸入是**逐字稿**，只負責逐字稿之後的後續處理。

```
[Windows]                          [WSL2 / Docker Desktop]
錄音 app (本專案)
  │ 原子化寫入
  ▼
D:\record ──▶ STT worker (輪詢 /mnt/d/record)
                  │ POST bytes (HTTP)
                  ▼
             whisper 容器 (hwdsl2/whisper-server)
                  │ return 逐字稿 JSON
                  ▼
             ~/stt-outbox (逐字稿資料夾)
                  │ openclaw 容器掛 WSL 原生資料夾、輪詢
                  ▼
             openclaw (後續處理平台，獨立)
                  │ POST messages (HTTP / OpenAI 相容)
                  ▼
             LLM 服務 (獨立容器) ── 內部跑 claude -p（訂閱）
                  │ return 整理後文字
                  ▼
             openclaw 寫 ~/openclaw-out、做摘要 / 分發
```

### 為什麼這樣切（取捨記錄）

**解耦點 1：STT 與 openclaw。**
- openclaw = 後續處理平台，天生輸入是**逐字稿**；要它去掃音檔、打 STT 是 scope creep。
- 「掃資料夾 → POST whisper → 寫逐字稿」是純 HTTP 工作，一支小 worker 即可，**不需要 openclaw**。
- 因為 worker 走 **HTTP 傳 bytes**（非掛載），所以**不需要** copier → ext4 → bind-mount → 處理 inotify 那一整套跨界複雜度。
- 紅利：STT 壞了不影響 openclaw；換 STT 後端不動 openclaw；想重轉某檔單獨跑 worker 即可。

**解耦點 2：openclaw 與 LLM 後端。**
- openclaw 與「呼叫 LLM 的程式」是**兩支獨立程式**，邊界是一個 **OpenAI 相容的 HTTP 合約**。
- openclaw 只認合約（送 messages、收 completion），**不知道後端是 `claude -p` 還是 API**。
- 紅利：未來 openclaw「改成 call API」= 改 LLM 服務內部、或 openclaw 直接改 `base_url`，**OpenClaw 呼叫碼一字不動**。跟 whisper 同一個 OpenAI 相容慣例，全系統風格一致。

> 已否決的方案：
> - **方案 B**（openclaw 容器直接掛 `/mnt/d`）：I/O 慢、inotify 不可靠、職責耦合。
> - **方案 A**（copier 複製到 ext4、openclaw 容器掛載掃音檔）：為了餵「容器化、掃資料夾的 openclaw」才需要的複雜度；既然 openclaw 改為只吃逐字稿，這套被整包移除。

## 3. 元件職責邊界（重要）

| 元件 | 位置 | 輸入 | 輸出 | 不負責 |
|------|------|------|------|--------|
| **錄音 app（本專案）** | Windows | 麥克風 / 系統音 | `D:\record` 的音檔（**原子化**） | STT、摘要、分發 |
| **STT worker** | Windows（detached） | `D:\record` 的音檔 | `~/stt-outbox` 的逐字稿 | 摘要、分發 |
| **whisper-server 容器** | Docker | HTTP 上傳的音檔 | 逐字稿 JSON | 檔案管理、輪詢 |
| **openclaw（後續處理平台）** | Docker（容器） | `~/stt-outbox` 的逐字稿 | `~/openclaw-out` + 摘要/分發 | 錄音、STT、LLM 後端細節 |
| **LLM 服務** | Docker（容器，獨立） | OpenAI 格式 messages（HTTP） | completion 文字 | 檔案、流程、輪詢 |

> 交接點：
> - 錄音 app ↔ STT worker：`D:\record`（檔案）
> - STT worker ↔ openclaw：`~/stt-outbox`（逐字稿資料夾，檔案）
> - **openclaw ↔ LLM 服務：`POST /v1/chat/completions`（HTTP，OpenAI 相容）** ← 可換後端的關鍵邊界

## 4. 技術選型（已查證）

### STT 模型服務：`hwdsl2/whisper-server`
- **作者**：Lin Song（GitHub `hwdsl2`，知名維護者，setup-ipsec-vpn 作者），**MIT 授權**，可信。
- **原始碼 repo**：`github.com/hwdsl2/docker-whisper`（注意：**沒有** `whisper-server` 這個 repo，那只是 image 名）。
- **底層**：faster-whisper，提供 **OpenAI 相容** `/v1/audio/transcriptions`。
- **對外 port**：9000；**模型快取 volume**：`/var/lib/whisper`。

| 環境變數 | 預設 | 本專案採用 |
|----------|------|-----------|
| `WHISPER_MODEL` | `base` | `large-v3-turbo`（中文準確度佳） |
| `WHISPER_DEVICE` | `cpu` | `cpu`（無 GPU）/ `cuda`（有 GPU） |
| `WHISPER_COMPUTE_TYPE` | int8(CPU)/float16(CUDA) | `int8`（CPU）/ `float16`（CUDA） |
| `WHISPER_THREADS` | `2` | **調成實體核心數**（CPU 吞吐關鍵） |
| `WHISPER_API_KEY` | 未設 | **建議設**（見安全注意事項） |
| `WHISPER_PORT` | `9000` | 9000 |

### LLM 後端：Claude（透過 `claude -p`，訂閱）
- **介面**：自建的 LLM 服務暴露 **OpenAI 相容** `/v1/chat/completions`；內部以 Claude Code 的 **headless 列印模式** `claude -p` 呼叫。
- **認證**：**訂閱**（非 API 計費）。`claude setup-token` 取得長效 token → 容器環境變數 `CLAUDE_CODE_OAUTH_TOKEN`。
- **模型**：`claude --model opus`（可換 `claude-opus-4-8` 等；最新最強的 Opus）。
- **為何包一層服務**：讓 openclaw 與後端解耦，未來可無痛換成直接打 API（見 §5.7）。

### 5.2 STT worker（`stt-worker/worker.py`，已實作骨架）

> **定案：worker 跑在 Windows**（不是 WSL）。讀 `D:\record` 原生最快；逐字稿寫到 **WSL 原生**
> 資料夾（`\\wsl.localhost\Ubuntu\home\<user>\stt-outbox`）供 openclaw 容器掛載。
> 原因見 §8「WSL 掛載雷區」—— 避開「Windows 資料夾→容器」這個不可靠模式。

一支輕量常駐程式，把「錄音檔 → 逐字稿」做掉：
- [x] **輪詢**（非 inotify）監看資料夾找新檔。
- [x] 只處理「完整」檔：略過 `.part`／點開頭檔 + 「大小連續 N 秒不變」雙保險。
- [x] 用 HTTP `POST` bytes 到 whisper 的 `/v1/audio/transcriptions`（OpenAI 相容），含重試+退避+逾時。
- [x] 逐字稿**原子化寫入** outbox（temp → `os.replace`）。
- [x] **冪等**：outbox 已有同名逐字稿就跳過，重啟不重跑；可選把來源 move 到 done 封存。
- [x] **單例守衛**：綁 `localhost:47654` 互斥鎖，同時只允許一支（重複啟動者自盡）。
- [ ] （可選）失敗檔毒丸標記、長錄音切段。

設定全走環境變數（見 `stt-worker/.env.example`）；由 app 啟動時自動帶入（§5.6）。

### 5.3 whisper-server 容器
參考啟動指令（已驗證，含建議調整）：
```bash

docker run -d \
  --name whisper-stt-server \
  --restart=always \
  --network audioflow-net \          # 與 STT worker 同網路；可不對 host 開 port

  -e WHISPER_DEVICE=cpu \
  -e WHISPER_COMPUTE_TYPE=int8 \

  -e WHISPER_API_KEY=<隨機字串> \    # ← 建議設定
```

### 5.4 openclaw（後續處理平台，獨立 — Docker 容器）

職責：以逐字稿為輸入，呼叫 LLM 做整理／摘要，產出可讀結果並分發。**檔案與流程歸它**。

- [ ] 容器掛 **WSL 原生** outbox：`-v /home/<user>/stt-outbox:/in`、`-v /home/<user>/openclaw-out:/out`（避開 §8 雷區）。
- [ ] **輪詢** `/in` 找新逐字稿（跨界事件不可靠，用輪詢）。
- [ ] 對每個新檔，以 **HTTP（OpenAI 相容）** 呼叫 LLM 服務 `POST /v1/chat/completions`（送 system + 逐字稿）。
- [ ] 取回整理後文字，**原子化寫入** `/out`（temp → rename），**冪等**（已產出同名就跳過）。
- [ ] 後續：摘要 / 分發（日曆、Discord、Obsidian…），屬 openclaw 範疇。

> **關鍵解耦**：openclaw 透過 OpenAI 相容 HTTP 呼叫 LLM，**不直接綁 `claude -p`**。未來改 call API，openclaw 呼叫碼不用動（見 §5.7）。
> 與 LLM 服務建議接同一 docker network，用容器名互連：`http://llm-service:8088/v1`。

### 5.5 Docker 把關 & STT 容器自動啟動（本專案，已實作）

**目標**：whisper 容器隨錄音 app 一併啟動；沒有 Docker Desktop 就不讓使用者錄音，並在首頁提示安裝。

> 註：錄音本身技術上不需要 Docker（只是寫音檔），但「不能轉譯就不給錄」是刻意的產品決策——避免錄了一堆無法進入後續流程的檔。

**後端**　`src/bridge/_docker.py`（`DockerMixin`，已併入 `Bridge`）
- `_docker_state()`：偵測 `not_installed` / `not_running` / `ok`，含 3 秒快取避免頻繁呼叫 docker CLI。
- `ensure_stt_container()`：容器不存在→`docker run` 建立（CPU 模式 `large-v3-turbo`+`int8`，threads 自動帶入核心數）；已停→`docker start`；用**非阻塞鎖**防定時重檢堆疊執行緒。
- `start_stt_async()`：背景執行緒，不卡 UI。
- `get_environment_status()`：給前端的狀態；**Docker 由關到開時自動重試**建立容器。
- Windows 下 `subprocess` 帶 `CREATE_NO_WINDOW`，避免閃主控台視窗。

**接線**
- `main.py`：`_on_started` 呼叫 `start_stt_async()` → app 啟動即拉容器。
- `_recording.py`：`start_recording()` 開頭把關，Docker 不可用回 `DOCKER_UNAVAILABLE`。
- `_helpers.py`：新增 `DOCKER_UNAVAILABLE` 錯誤型別。

**前端**　`src/frontend/index.html`
- 頂部 `#env-banner` 橫幅；`refreshEnvStatus()` 每 5 秒輪詢狀態。
- 沒裝 Docker → 紅色橫幅 +「前往安裝」連結、錄音鈕 disabled、`startRecording()` 再擋一層。
- 裝了沒開 → 黃色提示；就緒 → 橫幅消失、解禁（使用者中途開 Docker 可自動解除封鎖）。

**狀態對應**

| 情境 | `can_record` | 首頁表現 |
|------|:---:|------|
| 沒裝 Docker Desktop | ✗ | 紅色橫幅 +「前往安裝」、錄音鈕禁用 |
| 裝了但未啟動 | ✗ | 黃色提示「請先開啟 Docker Desktop」 |
| 就緒 | ✓ | 橫幅消失、可錄音；容器自動啟動 |

### 5.6 STT worker 生命週期（隨 app 啟動、獨立存活，已實作）

**目標**：worker 隨 app 啟動，但**關閉 app 後仍持續運作**（繼續轉譯積壓與新檔）。

> 決策：worker 為 **detached 背景程序**，非綁定子程序。重開機後不會自動啟動，須等下次開 app
> 才被拉起（符合「隨 app 啟動」語意）；若要開機常駐，改用 Windows 排程工作／服務。

**後端**　`src/bridge/_worker.py`（`WorkerMixin`，已併入 `Bridge`）
- `start_worker_detached()`：先用 `_worker_running()` 偵測；沒在跑才以 detached 程序拉起
  （Windows `DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP` → 脫離 app、不隨關閉而死）。
- 由 `config.json` 帶入環境：`STT_WATCH_DIR`（= `local_path`）、`STT_OUTBOX_DIR`（= `transcripts_path`）、
  `WHISPER_URL`、`STT_LANGUAGE=zh`、`STT_LOG_FILE`（detached 無主控台，log 寫 `stt-worker/worker.log`）。
- `get_worker_status()`：查詢 worker 是否在跑。

**去重 / 單例**　app 端偵測 + worker 端互斥鎖雙保險：
- worker 啟動綁 `localhost:47654`；被占用就自盡 → 同時只會有一支。
- app 連線該 port 判斷是否已在跑 → 不重複拉起。

**接線**
- `main.py`：`_on_started` 呼叫 `start_worker_detached()`。
- `config.json` + `_base.py`：新增 `storage.transcripts_path`（已填驗證過的 WSL 原生路徑）。

**生命週期對應**

| 動作 | 行為 |
|------|------|
| 開 app | worker 沒在跑→detached 拉起；已在跑→略過 |
| 關 app | **worker 繼續執行**，轉譯不中斷 |
| 再開 app | 偵測到 worker 存活→不重複啟動 |
| 重複啟動 worker | 撞鎖 port→立即結束 |
| 重開機 | 不自動啟動，待下次開 app |

**待辦**
- [ ] 在 Windows 實機驗證 detached 啟動 + 寫 UNC outbox（WSL 已驗證單例鎖與容器掛載讀取）。
- [ ] 如需「開機常駐」（不依賴開 app），另加 Windows 排程工作／服務。

### 5.7 LLM 服務（`llm-service/`，已實作 — 獨立容器）

一支獨立程式：對外 **OpenAI 相容** HTTP，對內以 `claude -p`（訂閱）為後端。**完全不碰檔案系統**，只做「messages 進 / completion 出」。供 openclaw（§5.4）透過 HTTP 呼叫。

**實作**　`llm-service/server.js`（純 Node，零依賴）+ `Dockerfile`（基底 `node:22-slim` + 裝 Claude Code）+ `README.md`
- 對外 `POST /v1/chat/completions`：解析 messages，system → `--append-system-prompt`、對話 → stdin 餵 `claude -p`，回傳包成 OpenAI 回應。
- `GET /health` 健康檢查。
- `--dangerously-skip-permissions`：純文字、零掛載，確保 headless 不卡權限。

**認證 / 設定**
- `CLAUDE_CODE_OAUTH_TOKEN`（訂閱 token，`claude setup-token` 取得）— **必填**，由 env 帶入（走訂閱、非 API 計費）。
- `CLAUDE_MODEL`（預設 `opus`）、`PORT`（預設 `8088`）。

**啟動**
```bash
docker build -t llm-service ./llm-service
docker run -d --name llm-service --network audioflow-net --env-file llm.env llm-service
```

**為什麼是 OpenAI 相容（而非簡單自訂端點）**
- 未來 openclaw「改 call API」可做到**連 LLM 服務都不用改** —— openclaw 把 `base_url` 指向真正的 API 即可。
- 跟 whisper 同一個慣例，全系統一致；openclaw 端可直接用任何 OpenAI 客戶端。

**待辦**
- [ ] 你本機跑 `claude setup-token`，token 放進 `llm-service/llm.env`（我無法代勞，需瀏覽器登入）。
- [ ] 多輪／長逐字稿的逾時與切段（目前一次一段，足夠整理用途）。

## 6. 資料夾與資料流

| 路徑 | 誰讀寫 | 用途 | 生命週期 |
|------|------|------|---------|
| `D:\record`（Windows） | 錄音 app 寫、worker 讀（皆 Windows 原生） | 錄音原始檔（封存） | 長期保留 |
| `~/stt-outbox`（WSL 原生 ext4）<br>Windows 視角 `\\wsl.localhost\Ubuntu\home\<user>\stt-outbox` | worker 寫（Windows→UNC）、openclaw 容器讀（掛原生） | 逐字稿 | 交給 openclaw 後可清 |
| `~/openclaw-out`（WSL 原生 ext4） | openclaw 容器寫 | 整理後 Markdown / 摘要 | 視分發策略 |
| `whisper-data`（Docker 具名 volume） | whisper 容器 | 模型快取 | 隨容器 |

> 注意：**LLM 服務不在此表** —— 它純走 HTTP（messages 進 / completion 出），**不碰任何資料夾**。
>
> 關鍵：全鏈**不出現「Windows 資料夾 → 容器」bind-mount**（§8 雷區）。
> 音檔走 HTTP 傳 bytes（whisper 容器零掛載）；逐字稿是 KB 級小檔，由 Windows 寫進 WSL 原生資料夾，
> openclaw 掛**原生 ext4** 讀取——此路徑已在實機驗證可行。

## 7. 里程碑（建議順序）

### M1：打通最小端到端（先求會動）
1. 錄音端**原子化落地**（5.1）。
2. 手動起 `whisper-server` 容器，用 `curl` 丟一個檔確認能回逐字稿。
3. 寫 STT worker（5.2）：讀一個檔 → POST → 寫出逐字稿。
> 完成標準：錄一段音，最後在 `~/stt-outbox` 看到對應逐字稿。

### M2：穩健化

## 8. 風險與注意事項

- ⚠️ **WSL 掛載雷區（本專案踩過）**：把 **Windows 資料夾 bind-mount 進容器**會出現「掛載訊息成功、實際讀不到（空目錄）」——成因多為 Docker Desktop 檔案分享未開、或 `/mnt/c`、`/mnt/d` 經 9P/drvfs 進容器後不可靠。**本架構刻意全程避開此模式**：音檔走 HTTP（whisper 零掛載）、逐字稿由 Windows 寫進 WSL 原生資料夾、openclaw 只掛**原生 ext4**。此路徑（Windows→`\\wsl.localhost`→容器掛原生）已實機驗證可讀寫雙向。
- ⚠️ **半寫入檔**：跨界讀檔最常見的坑。必靠錄音端原子化 + worker 的完整性判斷。
- ⚠️ **inotify 跨界失效**：`/mnt/d` 上檔案事件不可靠 → 監看 Windows fs 一律**用輪詢**。
- ⚠️ **CPU 效能**：`large-v3-turbo` + CPU 對長錄音非即時，批次可接受；有 NVIDIA GPU 則改 `:cuda` tag 大幅加速。
- ⚠️ **安全**：別把無認證的 9000 裸開到不受信任網路；設 `WHISPER_API_KEY`，或讓 worker 與 whisper 走 docker network。
- ⚠️ **供應鏈**：image **pin 具體版本**，勿用隱含 `latest`。
- ⚠️ **資源**：模型首次下載需外網；WSL2 需配足 RAM。
